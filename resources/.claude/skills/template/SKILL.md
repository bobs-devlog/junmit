# 회의 유형 가이드 생성/조정 워크플로우

"회의 유형" 관리 화면에서 진입하는 스킬입니다. **시작 입력은 `request.json` 파일**로 받고(셸 인자/환경변수 아님),
**결과는 staging에 기록**한 뒤 신호를 보냅니다. 그 뒤로는 **이 터미널에서 사용자와 대화하며 초안을 다듬습니다**
(`/assist`와 동일한 방식 — PTY를 살려두고 후속 요청을 받음). 앱이 결과를 미리보기로 보여주고, 사용자가
"저장"을 누르면 확정합니다. **저장/취소는 앱 UI가 처리하므로 이 스킬은 종료 신호(app_phase_done)를 보내지 않습니다.**

```bash
TEMPLATES_DIR="$HOME/Library/Application Support/app.junmit/templates"
STAGING="$TEMPLATES_DIR/.staging"
```

`$ARGUMENTS`는 사용하지 않습니다.

> **스크립트는 `$APP_DIR`(이 세션이 시작된 작업 루트 — `lib/`이 여기 있음)에 있습니다.** `signal.sh` 호출은 `$APP_DIR/lib/signal.sh` 절대경로로 하고 **절대 `cd` 하지 마세요.** 하네스가 "Base directory: …/skills/template"을 안내해도 스크립트는 거기 없습니다. 신호(`app_template_ready`)는 앱 미리보기 갱신의 **필수**라, `No such file`이 떠도 경로 오류이지 생략 사유가 아닙니다 — 임의로 건너뛰지 마세요.

---

## 0단계 — 요청 읽기

> **출력 원칙 (중요)**: 사용자가 알면 좋은 내용(파악한 유형·진행 방향·보강 계획 등)은 **중계해도 좋습니다** — 오해를 일찍 잡는 데 도움이 됩니다. 다만 **내부 구현·내부 점검 단계는 노출 금지**:
> - "규칙·예시 파일을 읽습니다", "기존 유형 규칙을 확인했습니다", **"컨텍스트를 (조용히) 확인하겠습니다"**, "TodoWrite가 없어서…" 같은 **내부 작업·메타 멘트는 출력 자체를 금지**. 그 단계를 했다고 알리지도, "조용히 한다"고 말하지도 마세요 — 그냥 다음 사용자 출력으로 넘어갑니다.
> - 도구명·파일명·`staging`·`frontmatter`·`slug` 같은 구현 용어 금지.
> - **기존 유형을 언급할 땐 slug(`note`/`review`/`presentation`/`retrospective`/`1on1`)가 아니라 표시 이름(`일반 회의`/`리뷰`/`발표/세미나`/`회고`/`1on1`)으로.** 사용자는 slug를 모릅니다. (각 유형의 표시 이름은 그 `.md`의 `label`)
> - **사용자에게 보내는 메시지는 평문으로** — `**굵게**`, 백틱, `#` 제목 같은 마크다운 강조 기호를 쓰지 마세요. 이 터미널은 그 기호를 그대로 글자로 보여줍니다 (예: `**'저장'**` → 화면에 별표가 그대로 노출). 강조하고 싶으면 따옴표('저장')나 문장으로.

`$STAGING/request.json`을 Read 합니다 (이 동작은 사용자에게 알리지 않음). 형태:

- **생성(create)**:
  ```json
  { "mode": "create", "name": "킥오프",
    "about": "새 프로젝트를 시작할 때 관련자들이 모여 배경과 목표를 맞추고, 범위와 역할 분담, 대략의 일정을 정함. 주관자가 간단한 자료를 준비해 오기도 함.",
    "output": "확정된 범위와 역할 분담, 첫 마일스톤",
    "existing": ["note", "review", "presentation", "retrospective", "1on1"] }
  ```
  각 필드는 사용자가 화면 폼에 입력한 값입니다. **사용자에게 언급할 땐 괄호 안 'UI 항목명'을 그대로** 쓰세요 (사용자가 자기가 본 항목을 알아보도록):
  - `name` (UI: **"유형 이름"**) = 표시명(한국어 가능) → 가이드의 `label`이 됩니다.
  - `about` (UI: **"이 회의는 어떤 회의인가요?"**) = 자유 서술. 여기서 목적·진행 구조·사전 자료 유무·특징적 발화 등 **의도 신호를 직접 추출**해 `summary`를 작성하세요.
  - `output` (UI: **"회의록에 꼭 담겼으면 하는 것 (선택)"**) = 비어 있을 수 있음. 값이 있으면 **본문 골격**(섹션·강조)에 반드시 반영. **비어 있어 "능동 보강"을 안내할 땐 이 항목명을 그대로 인용**하세요 (예: "'회의록에 꼭 담겼으면 하는 것'을 적지 않으셔서…").
  - `existing` = 이미 있는 유형 slug 목록 (충돌·중복 판단용. UI엔 안 보이는 내부 값이라 사용자에게 언급하지 마세요).
- **조정(adjust)**: `{ "mode": "adjust", "target": "note" }`
  - `target` = 조정 대상 유형의 `name`(slug). **무엇을 바꿀지는 request.json에 없고, 사용자가 터미널 대화로 알려줍니다.**

요청 파일이 없으면 "요청 정보를 찾을 수 없습니다"를 안내하고 종료합니다.

---

## 1단계 — 컨텍스트 로드 (생성·조정 공통) · **내부 단계 (사용자에게 중계 금지)**

> 이 단계는 사용자에게 **아무것도 출력하지 마세요**. "규칙/예시/기존 유형을 확인했다", "컨텍스트를 확인하겠습니다" 같은 멘트도 금지 — 바로 다음 사용자 출력(착수 안내·질문)으로 넘어갑니다.

- 공통 작성 규칙: `.claude/skills/meeting/notes-rules.md`
- 시드 5종을 **좋은 예시**로 참고: `$TEMPLATES_DIR/note.md`, `$TEMPLATES_DIR/review.md`, `$TEMPLATES_DIR/presentation.md`, `$TEMPLATES_DIR/retrospective.md`, `$TEMPLATES_DIR/1on1.md` (있는 것만). 각 파일의 `label`(예: note→"일반 회의")도 함께 파악해 두면, 사용자에게 기존 유형을 언급할 때 표시 이름으로 말할 수 있습니다.
- 기존 유형은 request.json의 `existing` 배열(slug)로 받음. 깊은 중복 판단이 필요하면 `$TEMPLATES_DIR/{slug}.md`를 Read해 `summary` 비교 (Read만, 파일 목록 명령 X)

## 2단계 — 첫 초안

### 생성(create)

**중복 점검 먼저**: `existing`와 요청을 비교해 **이미 충분히 커버되면** 새로 만들지 말고 안내 후 종료:

> "{유형 이름}" 요청은 기존 "{기존 label}"과 거의 같아 보입니다. 새로 만들기보다 그 유형을 "조정"하시길 권합니다.

(이때 staging에 쓰지 않음.)

**도메인 판단 + 능동 보강 (핵심 — 받아쓰기 금지)**: 사용자가 적은 것만 그대로 옮기지 마세요. `name`·`about`으로 **이 회의가 어떤 종류인지**(킥오프·고객 미팅·면접·워크숍 등) 파악하고, **그 유형에 일반적으로 유효한 구조·항목을 능동적으로 보강**하세요. 예:
- 킥오프 → `배경·목표` / `범위` / `역할 분담` / `마일스톤` 같은 표준 틀
- 고객 미팅 → `고객 요청·요구사항` / `우리 측 답변·제안` / `합의·후속`
- 면접 → `역량·항목별 평가` / `종합 의견` (단 세부 평가 루브릭은 조직마다 다르므로 일반 틀만; 구체 루브릭은 사용자에게)

원칙: 사용자의 `about`/`output` 의도가 **척추**, AI 보강은 그 의도를 더 잘 살리는 선에서. 보강은 **구조(섹션) 수준**이지 가짜 회의 내용 생성이 아닙니다 (예시 회의록의 내용은 자리표시자 유지).

**확인 질문은 꼭 필요할 때만** (`AskUserQuestion`, 터미널에 표시):
- **제목과 설명이 본질적으로 어긋날 때** — 예: 이름은 "고객 미팅"인데 `about`은 내부 팀 회의 묘사 → 추측으로 밀지 말고 **1문항**으로 방향 확정 ("외부 고객과의 회의로 만들까요, 설명대로 내부 회의로?").
- **구조를 좌우하는 핵심이 빠졌을 때만** 1개(많아야 2개) 질문.
- 그 외(어떤 섹션을 넣을지 같은 *열린 선택*)는 **묻지 말고 AI가 판단**해 초안부터 보여주고, 나머지는 사용자가 이후 **터미널 대화로 다듬게** 하세요. 질문 남발 금지 — 좋은 초안을 빨리 보여주는 게 우선.

**착수 안내 (권장)**: 초안을 쓰기 전에 파악한 내용을 **1~3줄로 짧게 중계**합니다 (사용자 언어로, 내부 구현 표현 없이). 예:

> 요청을 보면:
> - 유형: 킥오프 (새 프로젝트 시작 회의)
> - 배경·목표와 범위, 역할 분담을 정리하는 방향으로 가이드를 만들겠습니다.
> - "회의록에 꼭 담겼으면 하는 것"을 적지 않으셔서, 킥오프에 표준적으로 유효한 구조를 능동 보강하겠습니다.

(불일치가 있으면 이 중계 대신 **확인 질문**을 먼저 — 위 참고.)

그다음 변별되는 유형의 `{name}.md` 전체를 작성해 `$STAGING/result.md`에 Write. frontmatter는 정확히 이 형식 (게이트 검증):

```markdown
---
name: <ascii slug>          # 소문자·숫자·하이픈만. 한국어 label을 짧은 영어로 (예: 회고 → retrospective). existing과 충돌 시 -2 등으로 고유하게
label: <유형 이름>           # request.json의 name 그대로 (한국어 가능, 짧게)
description: <한 줄 설명>     # about을 한 줄로 요약 (UI 버튼 보조 텍스트)
title_keywords: <쉼표 구분>   # 선택 — 회의 **제목**에 들어가면 이 유형임이 명백한 단어만 (예: 회고, kpt). 자동 판별 1단계(결정론)가 사용. 명백한 단어가 없으면 이 줄 생략
summary: |
  <이 유형이 어떤 회의인지 1~2문장 (about에서 도출)>
  의도 신호: <about에서 읽어낸 구체 신호. 시드의 "의도 신호:" 줄 형식. 기존 유형과 변별되게>
---

# {유형 이름} 작성 가이드

> 공통 규칙은 `.claude/skills/meeting/notes-rules.md`를 먼저 적용하세요. 이 파일은 본문 골격만 정의합니다.

## 작성 형식
... (시드 구조 참고, 이 유형에 맞는 섹션·규칙. SPEAKER_XX 라벨 규약 유지. output 항목은 섹션·강조로 반드시 반영)

## 예시 회의록
... (markdown 코드펜스 안에 이 유형으로 나올 회의록 한 편의 샘플. 내용은 Bobs·SPEAKER_XX·{자리표시자}. 시드의 예시 참고)
```

> **`## 예시 회의록`은 필수** — 게이트가 검증하고 UI가 미리보기로 보여줍니다. 헤딩 + ```markdown 코드펜스 안에 샘플 회의록을 넣으세요.

### 조정(adjust)

`$TEMPLATES_DIR/{target}.md`를 Read해 현재 내용을 파악합니다 (이 동작·"읽었다/올렸다/확인하겠다" 류 멘트는 **출력하지 않음**). **이 시점엔 아무것도 쓰지 않습니다.** 현재본 미리보기는 앱이 이미 보여주고 있으니, staging에는 사용자의 변경 지시를 받은 뒤(4단계)에야 씁니다.

(대상 파일이 없으면 "대상 유형을 찾을 수 없습니다" 안내 후 종료.)

## 3단계 — 안내 (+ 생성은 신호)

- **생성**: `result.md`를 썼으니 신호를 보냅니다 → `bash -c 'source "$APP_DIR/lib/signal.sh" && app_template_ready'`. 그리고 초안을 만들었다고 알리되 **이 유형에 맞춰 어떤 표준 구조를 보강했는지 1줄로 짚어줍니다**. 예: "'킥오프' 초안을 만들었어요. 킥오프에 맞춰 '배경·목표 / 범위 / 역할 분담 / 마일스톤' 섹션을 넣었습니다. 왼쪽 미리보기를 확인하시고, 더 바꿀 점이 있으면 알려주세요. 마음에 들면 화면의 '저장'을 눌러주세요."
- **조정**: 아직 쓴 게 없으니 **신호를 보내지 않고**, 바로 안내만: "'{label}' 유형이에요. 지금은 이런 구조예요(1줄 요약). 어떻게 바꿀까요? (예: '결정사항을 표로', '액션 아이템에 기한 추가')"

그리고 **사용자 입력을 기다립니다** (claude 입력 bar 활용).

---

## 4단계 — 다듬기 대화 루프

사용자가 바꿀 점을 말할 때마다:

1. 기준 내용을 **전체 재작성**해 `$STAGING/result.md`에 Write (부분 Edit 금지 — frontmatter 형식 깨짐 방지). 기준은: 이미 result.md가 있으면 그것, **조정의 첫 변경이라 아직 없으면 조정 대상 원본(`{target}.md`)**.
   - frontmatter 형식 엄수: `name`/`label`/`description` 한 줄, `summary: |` 블록, `## 예시 회의록` 유지. 원본에 `title_keywords` 줄이 있으면 **그대로 유지**(조정 요청이 명시적으로 바꾸라는 경우만 수정) — 빠뜨리면 제목 기반 자동 판별에서 이 유형이 빠진다.
   - 예전 가이드라 `summary` 블록·`## 예시 회의록`이 없으면 이때 **형식을 보강**(게이트 통과 위해, 내용 구조는 유지).
   - **조정 모드에선 `name`(slug) 절대 변경 금지** (`{target}`와 동일 유지 — 바뀌면 원본이 남음).
   - 생성 모드에서 사용자가 이름을 바꾸길 원하면 `label`만 바꾸고 `name` slug는 가급적 유지 (이미 미리보기 중).
2. 신호 발신: `bash -c 'source "$APP_DIR/lib/signal.sh" && app_template_ready'`
3. "반영했어요. 더 바꿀 점 있나요? 없으면 '저장'을 눌러주세요." 정도로 짧게 안내하고 다시 입력 대기.
   - 이 유형에 흔히 유효한데 빠진 게 보이면 **제안 1가지**를 곁들여도 좋습니다(강요 금지, 예: "킥오프라면 '마일스톤에 목표 날짜'도 넣을 수 있는데 추가할까요?"). 매 턴 제안을 반복하지는 마세요.

사용자가 만족하면 앱에서 "저장"(또는 "취소")을 누르고, 그러면 앱이 이 터미널을 정리합니다. **스킬이 스스로 종료하거나 `app_phase_done`을 호출하지 않습니다.**

## 동작 원칙

- **출력·대화는 한국어 (첫 글자부터)** — `Let me…`/`Now I'll…` 류 영어 서두, "이제 ~하겠습니다"식 작업 메타 발화 금지. 내부 구현 용어(slug·frontmatter·staging·TodoWrite·파일 읽기 등) 사용자 노출 금지.
- **사용자 메시지는 평문** — 마크다운 강조(`**`, 백틱, `#`) 금지. 터미널에 기호가 그대로 보인다. 강조는 따옴표나 문장으로.
- **유의미한 중계는 OK, 내부 구현·메타 멘트는 금지** — 파악한 유형·진행 방향·보강 계획처럼 사용자가 알면 좋은 건 사용자 언어로 짧게 알려도 좋음. 단 도구·파일·내부 점검 단계는 **언급 자체를 안 함** ("규칙 확인했다", "컨텍스트를 (조용히) 확인하겠습니다" 류 멘트 금지).
- **기존 유형은 표시 이름으로만 언급** — slug(`note`/`review`/`presentation`/`retrospective`/`1on1`)를 사용자에게 노출하지 말 것. 표시 이름(`일반 회의`/`리뷰`/`발표/세미나`/`회고`/`1on1`, 각 `.md`의 `label`)으로.
- **결과는 항상 `$STAGING/result.md`에만 기록** — live `{name}.md`는 직접 건드리지 않음 (앱이 게이트 검증 후 확정).
- **매 변경마다 result.md 전체 재작성 + 신호 재발신** — 그래야 미리보기가 갱신됨.
- **frontmatter·예시 형식 엄수** — 어기면 저장이 거부됩니다.
- 중복 유형은 새로 만들기보다 조정 권고.
