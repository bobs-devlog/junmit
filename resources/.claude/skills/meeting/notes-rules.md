# 회의록 작성 공통 규칙

> 이 파일은 모든 회의 유형에 공통으로 적용되는 규칙입니다. 각 유형별 작성 형식은 `~/Library/Application Support/app.junmit/templates/{type}.md`를 참고하세요.
> 파일 책임·sidecar 분리·SPEAKER_XX sentinel·금지/허용 사항은 [SKILL.md](SKILL.md)에 정의되어 있습니다 — 이 가이드는 그 정책을 전제로 합니다.

---

## 회의 유형 결정

`$SESSION_DIR/meeting.json`을 읽고 `type` 필드를 확인하세요:

- **`auto` 또는 `type` 필드 없음** → 아래 "자동 판단" 절차로 적합 유형 결정 → **결정 결과를 `meeting.json.type`에 즉시 갱신** (한 번 판단된 결과 유지, 다음 재작성 시 자동 판단 반복 X). **`/meeting` 흐름에선 이 판단을 1단계의 `meeting-type-classification` sub-agent가 후보정과 병렬로 미리 수행**하므로, 메인은 그 보고의 `TYPE_DECISION:` 값을 적용만 하고 재분류하지 않습니다 (아래 "자동 판단"은 그 sub-agent가 따르는 동일 기준이자, 1단계를 skip한 경우(재작성·크래시 후 재시도로 `transcript_corrected.txt`가 이미 존재)나 명시 유형 파일이 삭제된 edge 케이스에서 메인이 인라인 fallback할 때의 절차)
- **`free-form`** → 자동 판단 skip + 아래 "Free-form 작성" 절차 직접 적용 (templates 매칭 X)
- **`presentation`, `note`, `review` 또는 사용자 정의 유형명** → `~/Library/Application Support/app.junmit/templates/{type}.md`를 Read. **그 파일이 없으면**(유형이 삭제·개명됨) → `auto`처럼 자동 판단 절차로 진행 + 결정 결과로 `type` 갱신 (작성 중단 금지)

### type 갱신 절차 (auto 케이스만)

회의 유형 결정 완료 직후 (회의록 작성 시작 전) `meeting.json.type`을 갱신하세요:
- templates 매칭 시: `type` = 결정된 templates id (예: `"note"`)
- free-form 결정 시: `type` = `"free-form"`

**❌ Edit 도구 사용 금지** — 단순 치환으로 형식이 안 깨지더라도 가이드 위반입니다. 다른 필드(`attendees` 등)에 같은 패턴이 적용될 때 깨질 위험을 원천 차단하기 위해 일관되게 금지합니다.

**✅ 패턴**: Read tool로 `$SESSION_DIR/meeting.json` 전체를 읽고, JSON parse 후 `type` 필드만 갱신해 Write tool로 전체 재작성하세요. 부분 치환이 아닌 전체 재작성이라 형식 깨짐 위험이 원천적으로 없습니다.

> 갱신 후 사용자가 회의록 탭의 유형 select에서 LLM 결정 결과를 즉시 인지. 다시 작성 시점에도 type이 그대로 적용되어 자동 판단 반복 없음. 사용자가 다시 LLM 판단을 원하면 select에서 명시적으로 "자동 판단" 다시 선택.

### templates 위치

회의 유형 가이드는 사용자 데이터 영역의 단일 위치에서 로드됩니다:

```
~/Library/Application Support/app.junmit/templates/
```

앱이 첫 실행 시 기본 가이드(presentation, note, review)를 자동 시드합니다. 사용자가 이 위치의 파일을 직접 편집하거나 새 `{name}.md`를 추가하면 자동 판단 후보에 포함됩니다.

### 자동 판단

위 디렉토리의 모든 `*.md` 파일을 순회하며 frontmatter의 `summary`를 수집하세요. 회의 내용(`transcript_corrected.txt` + `meeting.json`의 `agenda` 필드)과 각 summary를 매칭해 가장 적합한 유형을 결정합니다.

**판단 우선순위**: summary의 *의도 신호*(회의 목적·구조·사전 자료 유무)를 1순위로 보고, *발화 분포*(누가 얼마나 말했는지)는 보조 신호. 예: "한 명 70%+ 발화"는 presentation·review 모두 정상이라 단독으로 구분 신호가 아님 — `agenda`/transcript의 호명·자기소개·주제 컨텍스트로 의도를 파악하세요.

> `agenda`는 캘린더에서 가져온 회의 본문 + 사용자가 편집한 컨텍스트입니다 (비어있을 수 있음). 아젠다에 명확한 신호("X PRD 검토" 등)가 있으면 자동 판단의 신뢰도를 크게 높입니다. 단 [SKILL.md](SKILL.md)의 "회의에 등장하지 않은 정보를 회의록에 추가 금지" 정책은 그대로 적용 — agenda 내용을 회의록에 그대로 옮기지 마세요. 어디까지나 *맥락 이해*용입니다.
>
> **예외 — 외부 링크는 참고 자료 섹션에 한해 수용**: agenda에 포함된 URL(Confluence/문서/PRD 등)은 review의 `## 참고 문서`, note의 `## 관련 정보` 같은 *참고 자료 섹션*에 옮길 수 있습니다. 산문 본문(논의/피드백/발언/결정)에는 절대 옮기지 마세요.

판단 결과는 둘 중 하나:

1. **명확히 한 유형이 적합함** → 그 `{type}.md` 가이드 적용
2. **여러 유형이 비슷하게 적합하거나 어디에도 명확히 맞지 않음** → free-form 작성 (이 파일의 "Free-form 작성" 절차)

판단 근거를 한 줄로 출력하세요:

```
📄 회의 유형: 발표/세미나 (발표 주제 사전 정의 + Q&A 구조)
📄 회의 유형: free-form (양방향 1:1 대화, 정형 패턴 없음)
```

## 품질 경고 확인 (필수 선행)

회의록 작성 **직전**에 `$SESSION_DIR/speaker_mapping.json`의 `_quality_warning` 필드를 확인하세요.

- **필드 없음 (정상)** → 유형별 가이드 그대로 적용. 경고 블록 생략.
- **`severe_overmerge` 포함** → 유형에 따라 처리 다름:
  - **note 유형** (사람별 보고 패턴) — 한 SPEAKER 70%+면 진짜 over-merge일 가능성 높음. 다음 두 조치:
    1. **회의록 본문 최상단(메타 줄 위)에 경고 블록 삽입**:
       ```markdown
       > ⚠️ **화자 분리 품질 경고**: 이 녹음은 여러 화자가 하나의 SPEAKER 라벨로 합쳐져 있습니다. 회의록의 "누가 한 말인지"는 사용자 확인이 필요합니다.
       ```
    2. **본문에서 개별 발언자 귀속을 낮춤**:
       - 각 논의 주제의 "**SPEAKER_XX**: 내용" 불릿은 **확신 있는 항목만** 기록. 섞인 발언은 주제 단위 산문으로 통합
       - 예 ✅: `- 주요 논의: X 도구 작업 어려움. 향후 방향: PR 기반 문서 정리`
       - 예 ❌: `- **SPEAKER_03**: 지적` / `- **SPEAKER_05**: 반박` (SPEAKER 섞여있어 귀속 부정확)
  - **presentation/review 유형** — 발표자 한 명이 70%+ 발화하는 게 정상 패턴이라 over-merge 경고가 false positive일 가능성 큼:
    - 발표자(가장 많이 발화한 SPEAKER)의 발언량은 정상으로 간주
    - 단 발표자 외 SPEAKER들의 발언이 섞여 있는 정황(같은 SPEAKER에 1인칭 답변과 3인칭 호명이 모두 등장 등)이 있으면 그 부분만 보수적으로 처리 (의견 출처·피드백 발화자의 SPEAKER_XX 라벨은 확신 있는 것만)
    - 경고 블록 삽입 **불필요** (정상 회의 패턴)
  - **free-form** — note 유형과 동일하게 처리 (보수적)
- **`undermerge_suspected` 또는 `oversplit_suspected`만 있음** → 정상 작성. 경고 블록 불필요.
  - oversplit의 경우 speaker_mapping에서 같은 이름이 여러 SPEAKER에 매핑됐을 수 있으며, 앱이 표시 시점에 자동 치환하므로 회의록 자체엔 영향 없음

**경고 블록 삽입 위치 (severe_overmerge 시):**
- 본문 최상단 (메타 줄 위) — 회의 제목은 Header가 별도로 표시하므로 본문에 H1 없음. 경고 블록이 첫 콘텐츠

상세 규칙은 [SKILL.md](SKILL.md)의 "SPEAKER_XX sentinel의 적용 범위" 표를 참고하세요. 핵심 요약:

## 참석자 섹션 표기 (모든 유형 공통)

참석자 섹션은 **`meeting.json`의 `attendees` 필드를 단일 진실 원천**으로 작성하세요 — 캘린더 기반의 실제 참석 명단입니다. `speaker_mapping`의 미확인 SPEAKER는 발화 분리 오차로 매칭 못 한 화자일 뿐 실제 참석자가 아니므로 **참석자 섹션에 추가하지 않음** (대부분 attendees 중 한 명이 이름 매칭만 안 된 케이스).

`meeting.json.attendees`의 회사 멤버 이름에는 **`@` 접두**를 붙이세요. publish 스킬의 ADF 변환기가 `@firstName` 패턴을 mention 노드로 변환하기 위함 — `@` 없이 평문 이름만 적으면 Confluence 페이지에 mention 처리가 안 됩니다.

| 케이스 | 표기 | 예시 |
|---|---|---|
| 회사 멤버 (영문 first name) | `@Bobs` | `- 참석자: @Bobs, @Charlie` |
| 역할/직책 부가 정보 | `@Bobs (역할)` | `- @Bobs (PM), @Charlie (백엔드)` |
| 외부 참석자·고객 등 (사내 mention 불가능) | 평문 그대로 | `- 김철수 (외부)` |
| `attendees`가 빈 배열 | dash placeholder | `- 참석자: -` |

`attendees`가 빈 배열일 때는 줄을 생략하지 말고 `- 참석자: -`로 placeholder를 남기세요. 사용자가 캘린더 매칭 또는 AttendeeEditor 입력을 깜빡한 케이스가 가장 흔하므로, 빈 표기가 "여기 채워야 함" 신호 역할을 합니다.

근거:
- 회사 사람들의 Atlassian accountId는 `@firstName` 패턴 기준으로 publish 스킬이 일괄 lookup
- 외부 참석자는 사내 mention dict에 없으므로 `@`를 붙이면 변환 실패 후 plain text로 fallback되어 결과는 같으나, 변환 시도 자체가 노이즈이므로 명시적으로 평문 유지
- 미확인 SPEAKER는 **발화 헤더** (`**SPEAKER_06**: ...`)에서만 라벨 유지 — 참석자 섹션 X. 본문 작성 후 사용자가 매핑 추가하면 표시 시점에 자동 치환됨

## Action Items `@assignee` 분기 (모든 유형 공통)

| 패턴 | 표기 | 예시 |
|---|---|---|
| **1인칭 자기지정** ("내가/제가 ~ 하겠다") | `@SPEAKER_XX` (발화자 SPEAKER) | "제가 X PRD 작성하겠습니다" → `@SPEAKER_03` |
| **3인칭 명시 지시** ("A는 X가 해주세요" 또는 "X가 ~") | **산문에 명시된 이름 그대로** | "X PRD는 Bobs가 해주세요" → `@Bobs` |
| **역할만 결정 (담당자 미정)** | 역할 자연어 | "백엔드 쪽이 검토 필요" → `@백엔드` |

근거:
- 1인칭 케이스는 발화자 식별 = 담당자 식별이라 발화 주체 표기와 동일 (SPEAKER_XX)
- 3인칭 명시 케이스는 산문에 직접 등장한 이름 = 확실한 정보
- 매핑 변경 시 자연어 이름은 자동 치환되지 않으나 산문 이름과 동일 한계 (refine 스킬로 보정)

## 결론 태그 (이모지 + 라벨)

논의·논점에 결론·상태를 명시할 때 다음 라벨을 사용하세요:

- `⚡ **결정:**` — 팀의 행동/방향이 바뀌는 합의
- `→ **향후 방향:**` — 결론은 있지만 공식 결정은 아닌 경우
- `❓ **보류:**` — 의도적으로 미룬 경우
- `❓ **미결:**` — 합의에 도달하지 못한 경우

**`⚡ 결정` 판단 기준**: "이 결정을 모르는 팀원이 나중에 봤을 때, 자기 행동을 바꿔야 하는가?"
- 그렇다면 `⚡ **결정:**`
- 아니면 `→ **향후 방향:**`

**적용 위치**: 결론 태그는 **논의 흐름 안에서**(논점 아래 또는 발언 다음) 사용합니다. `## 결정`·`## 향후 일정` 같은 **요약 섹션의 불릿엔 태그를 붙이지 말고 평문으로** 작성하세요. 요약 섹션은 제목이 이미 성격을 드러내 태그가 군더더기이고, 발행 시 리스트 항목 안 패널로 변환돼 Confluence에서 편집 불가한 레거시 콘텐츠가 됩니다.

## Free-form 작성

자동 판단 결과 어느 유형에도 명확히 맞지 않을 때 사용합니다 (예: 1:1 양방향 대화, 비구조 토론, 인터뷰, 워크숍, 브레인스토밍).

### 작성 원칙

- **회의 성격에 맞춰 LLM이 섹션 구조 결정** — 정형 템플릿 강제 X
- **위 공통 규칙은 모두 적용**: SPEAKER_XX sentinel, Action Items 분기, 품질 경고 처리, 결론 태그
- **회의록 본문 파일**: `$SESSION_DIR/meeting-notes.md`
- **시작은 자연스러운 형태로**: 메타(날짜·참석자) + 본문 섹션. 본문 섹션 헤딩과 깊이는 회의 흐름에 맞춰 자유롭게. **본문 첫 H1(회의 제목)은 작성하지 않음** — 회의 제목은 `meeting.json.title` 단일 진실 원천에서 가져오며, 앱이 Header에 표시하고 publish 스킬이 Confluence page title로 사용

### 권장 골격 (참고용 — 회의 성격에 맞춰 변형)

```markdown
- 날짜: {날짜}
- 참석자: @{firstName1}, @{firstName2}, ...    ← 회사 멤버는 @firstName 접두

## {회의 성격에 맞는 섹션 1}

{회의 흐름에 맞춘 산문 또는 발언 불릿}

## {회의 성격에 맞는 섹션 2}
...

## Action Items (있는 경우만)
- [ ] {task} @{담당자}
```

### 발언 기록 형식 (선택)

발언을 기록할 필요가 있으면 다음 형식 중 회의 성격에 맞는 것을 선택:

- **불릿 나열**: `- **SPEAKER_03**: 발언 내용`
- **Q&A 흐름**: 들여쓰기 + `→` 또는 `**SPEAKER_05**: 답변`
- **주제별 통합 산문**: 발언을 의견 단위로 통합 (특정 SPEAKER 귀속 부담스러우면)

발언별 SPEAKER 귀속이 정확하다는 확신이 있을 때만 SPEAKER_XX 라벨을 박으세요. 확신 없으면 산문으로 통합.
