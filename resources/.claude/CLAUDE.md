# 스킬 실행 가이드

`.claude/skills/`의 스킬들이 PTY에서 실행될 때 적용되는 공통 규칙입니다.
사용자(비개발자 포함)가 터미널 출력을 직접 보는 환경입니다.

## 사용자 친화 출력 규칙

다음 규칙을 모든 스킬에 공통 적용:

- **첫 글자부터 한국어** — 스킬 실행의 **맨 첫 출력부터** 한국어로 시작합니다. `Let me…`/`Now I'll…` 같은 영어 서두 금지.
- **작업 시작·전환 메타 발화 금지** — "이제 ~를 하겠습니다", "먼저 ~를 확인합니다", "X 파일이 없으므로 ~", "~를 분석한 뒤 ~하겠습니다" 같은 **내부 진행 설명을 산문으로 출력하지 않습니다**. 진행 상황은 오직 `TodoWrite`로만 표시하고, 사용자에게 내는 텍스트는 각 SKILL.md가 정의한 한국어 결과 요약(`🎤 …`, `📄 …`, `✅ …` 등)뿐입니다. 사용자에게 보이는 **첫 텍스트 출력은 작업 결과 요약부터** 시작해야 합니다.
- **이 금지는 첫 출력만이 아니라 작업 전 구간에 적용** (중요) — 특히 **도구(`Bash`·sub-agent 등) 호출 직전·직후에 "무엇을 할지/했는지" 중계하는 짧은 발화를 출력하지 마라**. 도구는 **말없이** 실행하고, 사용자에게 보이는 텍스트는 SKILL.md가 정의한 결과 요약 블록뿐이다. 단계 전환 시에도 "다음으로 ~합니다" 같은 연결 멘트를 넣지 마라 — `TodoWrite` 마킹이 그 역할을 한다.
  - ❌ 실제로 샌 예: `"All 3 sub-agents done. Now create corrected.txt and apply edits via sidecar."` / `"Now let me load the type guide (note) and the notes-rules, then write the meeting notes."` / `"먼저 ~를 정리합니다"`
  - ✅ 올바른 거동: 이런 전환 중계를 **아예 출력하지 않는다**. 도구를 바로 실행하고, 단계 결과가 나오면 그때 `🎤`·`📝`·`📄`·`✅` 요약 블록만 출력한다.
- **모든 출력과 대화는 한국어로 진행**
- **내부 지침 용어를 사용자 출력에 복사 금지** (핵심) — SKILL.md·이 파일의 본문에 적힌 용어는 **너에게 내리는 내부 설명**이다. 그 용어를 사용자 출력에 그대로 옮기지 마라. 아래 매핑표의 **왼쪽은 지침을 읽을 때만** 쓰고, 사용자에게는 **오른쪽(또는 출력 안 함)**으로 표현한다:

  | 내부 지침 용어 (출력 금지) | 사용자 출력 |
  |---|---|
  | `교정 포함/미포함` / `detailed_correction` | "전사본 교정" — 굳이 안 내는 게 기본 |
  | `Phase` / `Phase-1` / `1단계`·`1·2단계`·`3·4단계` 등 **단계 번호** | **출력 금지** — 진행은 `TodoWrite`로만 |
  | `transcript.txt`·`transcript_corrected.txt`·`meeting.json`·`speaker_mapping.json` 등 **모든 파일명** | 노출 금지 ("전사본"·"교정본"·"회의 정보" 등 일반어) |
  | `type=note`·`type` 필드·`auto`·`free-form` | 노출 금지 (유형은 한국어 표시 이름 "일반 회의"·"리뷰" 등으로) |
  | `sub-agent` / `spawn` / `sidecar` / `sentinel` | "AI 다듬기"·"AI가 정리"처럼 사용자 시점 표현 |
  | `TYPE_DECISION` / `_quality_warning` 등 코드 식별자 | 노출 금지 |

  - ❌ 출력 예: `"Let me set up progress tracking and begin Phase 1. (no transcript_corrected.txt), 정밀 경로 (detailed_correction: true), type=note (분류 sub-agent 불필요)."`
  - ✅ 올바른 거동: **이런 메타 발화를 아예 출력하지 않는다.** 진행은 `TodoWrite`로 보이고, 첫 텍스트 출력은 결과 요약(`🎤 화자 매칭 …`)부터.

- **진행 상황은 `TodoWrite`로 표시** — 초기화 시점에 **첫 항목을 `in_progress`, 나머지를 `pending`**. 사용자가 어느 단계가 진행 중인지 즉시 시각적으로 확인할 수 있어야 함. 모두 pending으로 초기화한 뒤 작업이 끝나서야 마킹을 한꺼번에 업데이트하면 진행 표시 의미가 사라짐
- **sub-agent 위임은 메인 컨버세이션 todos에서 한 항목으로 묶음** — sub-agent 자체 TodoWrite는 메인과 분리되므로, 메인은 sub-agent 결과를 받은 뒤 자체 마킹만 업데이트

단계 매핑 표·스킬 특화 분기(재작성 모드 등)·단계별 한국어 요약 출력 형식은 각 SKILL.md에서 정의합니다. 그 본문에 등장하는 단계 번호·파일명·분기 용어는 **모두 내부 지침**이며 위 규칙에 따라 사용자 출력에 노출하지 않습니다.

## 세션 파일 수정 공통 규칙

스킬 워크플로우 진행 중이든, 워크플로우 종료 후 같은 PTY에서 이어지는 사용자 추가 요청이든 — **어느 시점이든** 세션 파일(`meeting-notes.md`·`speaker_mapping.json`·`meeting.json` 등)을 수정했다면 다음 두 규칙을 적용합니다:

1. **수정 후 매번 UI 갱신 신호 (필수)** — 앱은 파일 변경을 스스로 감지하지 못합니다. 이 신호가 없으면 사용자 화면에 수정 결과가 반영되지 않습니다:

   ```bash
   bash -c 'source "$APP_DIR/lib/signal.sh" && app_refresh'
   ```

   `$APP_DIR`은 PTY spawn 시 전달된 작업 루트 환경변수입니다. **절대 `cd` 하지 말고** 이 형태 그대로 호출하세요. `No such file`이 떠도 경로 오류이지 생략 사유가 아닙니다.

   **면제**: 수정 직후에 어차피 phase 신호(`app_phase_step_done`·`app_phase_done`)를 보내는 워크플로우 단계라면 그 신호가 화면 갱신을 겸하므로 `app_refresh`를 따로 보내지 않습니다. 즉 이 규칙의 주 대상은 **phase 신호가 없는 수정** — 워크플로우 종료 후 추가 요청, assist류 자유 작업 — 입니다.

2. **대규모 수정 전 백업** — 회의록 본문의 섹션 재작성·축약·확장·유형 변경·톤 변경 등 큰 변경 전엔 백업을 먼저 만듭니다:

   ```bash
   cp "$APP_SESSION_DIR/meeting-notes.md" "$APP_SESSION_DIR/meeting-notes.bak.$(date +%Y%m%d_%H%M%S).md"
   ```

   오타 정정·항목 1개 추가/삭제·문장 다듬기 같은 작은 수정은 백업을 생략합니다 (`.bak` 파일 누적 방지). 판단 기준은 하나 — "사용자가 복구하고 싶을 변경인가?"
