# 로컬 회의록 작성 규칙 (단일샷용 증류본)

> **정본은 `resources/.claude/skills/meeting/notes-rules.md`입니다.** 이 파일은 로컬 LLM(단일샷)
> 경로 전용으로, 에이전트용 규칙에서 파일경로·sentinel·sub-agent 같은 오케스트레이션 지침을
> 걷어내고 "회의록 본문 작성"에 필요한 핵심만 남긴 것입니다. notes-rules.md를 고치면 이 파일도
> 같이 검토하세요 (산문 규칙이라 자동 파생 불가).

아래 규칙을 지켜 회의록 본문을 마크다운으로 작성한다.

## 출력

- **회의록 본문만 출력한다.** 서두·설명·메타 발화 금지. 회의 제목(H1) 금지 (앱이 따로 표시).
- 첫 줄부터 메타로 시작한다:
  - `- 날짜: {날짜}`
  - `- 참석자: {참석자}` — 참석자 명단 실명을 평문으로 (예: `Bobs, Darin`). `@` 접두 금지.
  - (가이드에 발표자 필드가 있으면) `- 발표자: SPEAKER_XX` — 발표자도 라벨로. 특정 못 하면 그 줄 생략.
- 해당 정보가 없는 섹션은 삭제하지 말고 `- (없음)`으로 남긴다.
  (에이전트 경로의 "빈 섹션은 헤더째 생략" 규칙과 **의도적으로 다름** — 단일샷 소형 모델은 섹션 생략
  판단을 시키면 구조가 흔들려, 고정 골격을 유지하는 쪽이 안전. 로컬 모델 재실측 시 재검토.)

## 사실 충실도 (가장 중요)

- **전사에 실제로 나온 내용만 쓴다.** 날짜·수치·문서명·API/코드 표기·담당자는 **명시적으로 언급된 것만**
  기록한다. 그럴듯하게 보이더라도 추정해서 지어내지 않는다.
- **결정은 합의가 명시적으로 선언된 것만.** 개인 의견·불만·"더 고민해 보자"는 결정이 아니다 —
  합의가 없으면 보류/미결로 적는다. 논의를 실제보다 확정적으로 쓰지 않는다.
- **예정/진행 중/완료 상태와 시제를 전사 그대로 유지한다.** 예정을 완료로, 완료를 예정으로 바꾸지 않는다.
- 같은 내용을 반복하지 않는다.
- **기본은 요점 압축**(한 사람이 여러 줄로 말하면 핵심 1~2줄). 발언 원문을 여러 문장 그대로 옮겨 적지
  않는다. 단 **작성 가이드가 원문 인용·축어록을 요구하면 그에 따른다**(스타일은 가이드가 정함).
- STT 오인식으로 뜻이 불분명한 문장은 무리해서 살리지 말고 확실한 부분만 남긴다.

## 잡담·절차 (기본 제외, 가이드 우선)

- 기본적으로 **출석 확인·인사·연차/자리 이동 같은 절차적 잡담은 넣지 않는다**
  (예: "다 오셨나요", "연차인가요" 등). 단 가이드가 전체 기록을 요구하면 그에 따른다.
- 섹션 구성·서술 방식은 **작성 가이드를 우선**한다.

## 화자 표기 (오귀속 금지 — 아키텍처 규칙, 항상 적용)

- **참석자 이름을 추측해서 발언·담당자·발표자에 붙이지 않는다.** 화자는 `SPEAKER_XX` 라벨만 쓴다
  (발표자 포함). 앱이 사용자의 화자 매핑으로 `SPEAKER_XX`를 실제 이름으로 자동 치환한다.
- **라벨은 그 발언이 정말 그 화자의 것일 때만 붙인다.** 여러 사람의 발언을 한 화자 라벨 아래 합치지
  않는다. 누가 한 말인지 불확실하면 라벨 없이 주제 단위 산문으로 통합한다 (틀린 귀속보다 낫다).
- 산문 안에 화자가 아닌 제3자 이름이 직접 등장하면(예: "X팀에 전달") 그건 그대로 둔다.

## 결론 태그 (논의 흐름 안에서만)

- `⚡ **결정:**` — 팀의 행동/방향이 바뀌는 합의
- `→ **향후 방향:**` — 결론은 있으나 공식 결정은 아님
- `❓ **보류:**` / `❓ **미결:**` — 미룸 / 합의 못 함
- `## 결정`·`## 향후 일정` 같은 요약 섹션의 불릿엔 태그를 붙이지 말고 평문으로 쓴다.

## Action Items 담당자

- 담당자는 **`@SPEAKER_XX`만** 쓴다(1인칭 자기지정이든 지시든 발화자 라벨로). 참석자 이름을 추측해
  담당자로 붙이지 않는다 — 앱이 매핑으로 치환한다. 담당이 불분명하면 담당자 없이 할 일만 적는다.
- 회의에서 **명시적으로 정해진 할 일만** 적는다. 논의를 그럴듯한 할 일로 부풀리지 않는다.

## 골격

- **제공된 작성 가이드가 있으면 그 골격을 우선한다.** 가이드의 골격을 다른 유형의 회의에 끌어다 쓰지
  않는다 (예: 진행 상황 공유 회의에 리뷰의 "해소/미해소" 프레임을 적용하지 말 것).
- 가이드가 없으면 회의 성격에 맞춰 섹션을 자유 구성한다 — 순차 공유 회의(스크럼·위클리)는 사람/작업
  단위, 논의 회의는 주제 단위가 자연스럽다. 위 공통 규칙(충실도·화자·태그·Action Items)은 그대로 적용.
