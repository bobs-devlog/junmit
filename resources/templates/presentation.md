# 발표/세미나 작성 가이드

> 공통 규칙은 `.claude/skills/meeting/notes-rules.md`를 먼저 적용하세요. 이 파일은 발표/세미나 형식의 본문 골격만 정의합니다.

---

## 무엇을 담나

발표/세미나 회의록은 *그 자리에 없던 사람*이나 *나중에 다시 보는 참석자*가 이해할 수 있게 다음을 담습니다 — **무엇이 공유됐나(핵심 내용) → 무엇을 묻고 답했나(Q&A) → (있으면) 논의·결정 → 더 볼 자료.** 메타를 제외한 모든 섹션은 **있을 때만** 작성하고 빈 섹션은 생략합니다 (`notes-rules.md`의 "빈 섹션 처리" 참고).

---

## 작성 형식

회의록 본문 파일은 `$SESSION_DIR/meeting-notes.md`. 다음 골격을 따르세요:

```markdown
- 날짜: {날짜}
- 참석자: {이름1}, {이름2}, ...    ← meeting.json attendees 기준 평문. 미확인 SPEAKER 라벨은 넣지 않음 (notes-rules.md "참석자 섹션 표기")
- 발표자: {매핑 확정 시 자연어 이름, 미확정이면 "(미확인)"}    ← 발표자가 여럿이면 이 줄 대신 각 ## 주제 아래에 표기

## {발표/주제}

### 핵심 내용
- {발표 요지 1 — 핵심 메시지·결론}
- {발표 요지 2}
- {발표 요지 3}

### Q&A
- **SPEAKER_XX**: {질문}
  - **→ SPEAKER_XX**: {답변 핵심}    ← 답변자가 발표자여도 발언 헤더는 SPEAKER 라벨 (앱이 표시 시점에 이름 치환)
- **SPEAKER_XX**: {질문}
  - **→ SPEAKER_XX**: {답변 핵심}

### 논의            ← Q&A를 넘어선 토론·이견이 있을 때만
- {핵심 의견 1-2줄} ({SPEAKER_XX})

→ **향후 방향:** {내용}

## 결정             ← 결정이 있을 때만. 불릿은 평문 (인라인 결론 태그 X)
- {decision}

## Action Items     ← 후속 작업이 있을 때만
- [ ] {task} @{담당자}

## 참고 자료         ← 공유된 자료가 있을 때만
- {슬라이드/문서 링크}
```

발표/주제가 여럿이면 `## {주제}`를 여러 번 쓰고 각 아래에 발표자·핵심 내용·Q&A를 둡니다. 주제 사이의 자유 토론은 `## 자유 논의` 같은 별도 섹션으로.

---

## 작성 원칙

- **핵심 내용 (요지)**: 발표에서 공유된 핵심 메시지·결론을 **불릿 3~5개**로 남깁니다. *발표 전체를 받아쓰거나 슬라이드를 재현하지 마세요* — 슬라이드는 `## 참고 자료`에 링크하고, 회의록엔 **나중에 기억할 요점**만. 자료가 따로 있어도 요지는 남기는 게 durable 기록으로 유용합니다. 발화에 없는 내용을 지어내지 말 것.
- **Q&A**: 발표/세미나의 정상 구성 요소입니다. 의미 있는 질문과 답변을 기록하되 *질문이 드러낸 관심사*와 *답변의 핵심*에 집중하고, 한담성 질문은 생략합니다. 형식은 `**SPEAKER_XX**: 질문` + 들여쓴 `**→ SPEAKER_XX**: 답변`. **답변자가 발표자로 확정되어 있어도 발언 헤더는 SPEAKER 라벨을 유지**합니다 (발언 헤더 실명 금지 — `.claude/skills/meeting/SKILL.md` sentinel 규칙. 발표자 이름은 메타의 발표자 줄로 충분). 질문자가 불분명하면 SPEAKER 라벨 없이 질문만 적어도 됩니다.
- **논의**: Q&A를 넘어선 토론·이견·방향 합의가 있을 때만. 논점 단위로 묶고 의견은 1-2줄 `(SPEAKER_XX)`. 각 논의에 결론 태그(`.claude/skills/meeting/notes-rules.md`의 "결론 태그" 참고) — 단 논의 흐름 안에서만 쓰고, `## 결정` 같은 요약 섹션 불릿엔 붙이지 않습니다.
- **결정 / Action Items**: 발표·세미나는 결정·후속이 없는 경우가 많습니다 → 없으면 섹션 자체 생략. 추측 금지. `@assignee` 분기는 `.claude/skills/meeting/notes-rules.md` 참고.
- **참고 자료**: 공유된 슬라이드·문서 링크. `agenda`에 포함된 URL도 본문이 아니라 이 섹션으로 옮길 수 있습니다 (notes-rules.md 참고).

---

## 발표자 줄 처리

발표자 매핑이 확정되어 있으면 자연어 이름(`- 발표자: Bobs`), 미확정이면 `- 발표자: (미확인)`. 사용자가 이후 매핑 추가하고 본문 보정으로 업데이트할 수 있습니다. 발표자 줄은 그룹핑/메타 표기라 자연어 이름이 허용되지만, **본문 발언 헤더(Q&A 답변 포함)는 항상 SPEAKER 라벨**입니다. 자세한 규칙은 `.claude/skills/meeting/SKILL.md`의 "그룹핑/메타 표기 처리 규칙" 표 참고.

## 예시 회의록

> 이 유형으로 만들어지는 회의록 샘플입니다 (형식 참고용 — 실제 작성 시 회의 내용으로 채웁니다. SPEAKER_XX 라벨은 앱이 화자 매핑에 따라 실제 이름으로 표시).

```markdown
- 날짜: 2026-06-05
- 참석자: Bobs, Charlie, Dana
- 발표자: Bobs

## 효과적인 회의록 작성법

### 핵심 내용
- 회의록의 목적은 "그 자리에 없던 사람이 결론과 맥락을 이해하는 것"
- 발언 순서가 아니라 논점 단위로 정리하면 가독성이 크게 오름
- 결정·후속 작업은 본문과 분리해 한곳에 모아야 추적이 쉬움

### Q&A
- **SPEAKER_01**: 회의 중 실시간으로 받아치면 정리가 안 되는데 어떻게 하나?
  - **→ SPEAKER_02**: 회의 중엔 키워드만 적고, 정리는 끝난 직후 한 번에 하는 방식을 권장
- **SPEAKER_04**: 자동 전사를 쓰면 받아쓰기는 필요 없지 않나?
  - **→ SPEAKER_02**: 전사는 원천일 뿐, 논점으로 재구성하는 건 사람이 해야 가치가 생김

### 논의
- 팀 공통 템플릿을 둘지에 대한 의견 교환 (SPEAKER_01)

→ **향후 방향:** 팀 위키에 회의록 템플릿 초안을 올려 의견 수렴

## Action Items
- [ ] 회의록 템플릿 초안 작성 후 위키 공유 @SPEAKER_02

## 참고 자료
- 발표 슬라이드 링크
```
