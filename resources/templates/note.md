# 회의록 작성 가이드 (note)

> 공통 규칙은 `.claude/skills/meeting/notes-rules.md`를 먼저 적용하세요. 이 파일은 회의록 형식의 본문 골격만 정의합니다.

---

## 무엇을 담나

일반 회의록은 *그 자리에 없던 사람*이 논의의 결론과 맥락을 따라갈 수 있게 담습니다 — **주제별로 무엇이 논의되고 어떤 결론이 났나 → 무엇이 결정됐나 → 누가 무엇을 하기로 했나.** 메타와 `## 논의`를 제외한 섹션(`## 목표`·`## 공지 · 일정`·`## 결정`·`## Action Items`·`## 관련 정보`)은 **있을 때만** 작성하고 빈 섹션은 생략합니다 (`notes-rules.md`의 "빈 섹션 처리" 참고).

---

## 작성 형식

회의록 본문 파일은 `$SESSION_DIR/meeting-notes.md`. 다음 골격을 따르세요:

```markdown
- 날짜: {날짜}
- 참석자: {이름1}, {이름2}, ...    ← meeting.json attendees 기준 평문. 미확인 SPEAKER 라벨은 넣지 않음 (notes-rules.md "참석자 섹션 표기")

## 목표             ← 회의 목적이 발화로 명시된 경우만 (없으면 섹션 생략 — agenda로 지어내지 말 것)
- {goal}

## 공지 · 일정       ← 일정·행사·인사 등 전달 사항이 있을 때만 (정기 회의에 흔함). 논의 없는 단순 전달은 논의 주제로 만들지 말고 여기에
- {announcement}

## 논의

### 1. {topic}

- {현황·공유 사항은 화자 없이 요지 불릿으로 통합}
- **SPEAKER_XX**: {이견·제안 등 "누가 말했는지"가 정보인 발언}
  - **→ SPEAKER_XX**: {응답}

→ **향후 방향:** {논의의 결론·상태 — "결론·결정 처리" 참고}

> {note — 비고가 있는 경우}

### 2. {topic}
...

## 결정             ← 결정이 있을 때만. 불릿은 평문 (인라인 결론 태그 X)
- {decision}

## Action Items     ← 후속 작업이 있을 때만
- [ ] {task} @{담당자} (기한: {due})

## 관련 정보         ← 공유된 링크·자료가 있을 때만
- {reference}
```

---

## 발언 통합 원칙 (받아쓰기 금지)

발언 불릿은 전사 재현이 아니라 **논지 요약**입니다:

- 같은 화자가 여러 줄에 걸쳐 말한 내용은 **핵심 1~2줄로 통합**
- 개별 발언 불릿(`**SPEAKER_XX**: ...`)은 *누가 말했는지가 정보일 때만* — 이견·제안·질문과 응답·결정 발의 등
- 단순 정보 공유·현황 보고는 화자 라벨 없이 주제 아래 요지 불릿으로 통합
- 각 주제가 발언 불릿 5개를 넘어가면 받아쓰기 신호 — 논점 단위로 다시 묶기

---

## 사람별 보고 vs 주제별 논의 구분

### 순차 발표형 (Daily Scrum, 위클리 미팅 등 "진행자 호명 → 각자 진행 사항 발표" 패턴)

- **사람마다 별도 주제(`###`)로 분리하지 말 것** — 한 회의록에 8~10개 사람별 주제가 나오면 가독성 떨어짐
- **권장 1**: 의미 있는 작업/프로젝트 단위로 묶기
  - 예: `### 프로젝트 X` 안에서 `- **SPEAKER_03**: 1차 PR 머지` / `- **SPEAKER_07**: 리뷰 진행 중`
- **권장 2**: 묶을 단위가 마땅치 않으면 하나의 통합 주제(예: `### 주간 진행 보고`) 안에서 사람별 sub-section(`####`)으로 분리하거나, 그냥 SPEAKER 발언 불릿 나열
- **사람별 sub-section 헤더 표기** (권장 2의 sub-section 사용 시):
  - 매핑 확정: 자연어 이름만 — `#### Bobs`
  - 매핑 미확정: SPEAKER 라벨 + 역할/맥락 — `#### SPEAKER_06 (백엔드)` (사용자가 매핑 추가하면 자동 치환되어 자연스럽게 채워짐)
  - sub-section은 발언 단위가 아니라 "한 섹션 전체가 누구의 보고인지" 결정한 결과물이라, 매핑 확정 시 SPEAKER sentinel 박지 않음
- 진행자의 호명 멘트("다음 누구 발표 부탁드려요")는 회의록에 기록하지 않음 — 회의록 본문에 별도 "진행자" 줄도 작성하지 않음

### 자유 토론형 (비정기 논의, 브레인스토밍)

- 주제별로 분리. 각 주제 안에서 위 "발언 통합 원칙"대로 요지 통합 + 필요한 발언만 불릿

---

## 논의 포맷 규칙

- 각 논의 주제를 `###` 헤딩으로 구분
- **발언 헤더는 매핑 확정 여부와 무관하게 `**SPEAKER_XX**:` 라벨** — 앱이 표시 시점에 실제 이름으로 치환합니다. 실제 이름(`**Bobs**:`)이나 "참석자A" 같은 임시 이름을 발언 헤더에 직접 박지 마세요 (sub-section 헤더만 예외 — 위 "사람별 sub-section 헤더 표기")
- Q&A 흐름은 들여쓰기 + `→`로 응답 표시
- 비고는 인용문(`>`)으로 주제 아래에 추가

---

## 결론·결정 처리

각 논의 주제 끝에 결론·상태를 결론 태그로 명시 (`.claude/skills/meeting/notes-rules.md`의 "결론 태그" 참고 — `⚡ 결정` / `→ 향후 방향` / `❓ 보류·미결`). 회의 전체의 핵심 결정은 별도 `## 결정` 섹션에도 평문으로 정리.

---

## Action Items 처리

분기 규칙은 `.claude/skills/meeting/notes-rules.md`의 "Action Items `@assignee` 분기" 참고. 회의록 유형은 기한(due)이 명시되는 경우가 많으므로 `(기한: {due})` 형식 활용.

---

## 예시 회의록

> 이 유형으로 만들어지는 회의록 샘플입니다 (형식 참고용 — 실제 작성 시 회의 내용으로 채웁니다. SPEAKER_XX 라벨은 앱이 화자 매핑에 따라 실제 이름으로 표시).

```markdown
- 날짜: 2026-06-05
- 참석자: Bobs, Charlie, Dana

## 논의

### 1. 신규 기능 출시 준비
- 1차 작업 완료, 다음 주 QA 예정
- **SPEAKER_03**: 일부 화면에서 예외 케이스 발견
  - **→ SPEAKER_01**: 별도 항목으로 분리해 처리하겠다고 답변

⚡ **결정:** 출시 전 발견한 예외 케이스는 별도 항목으로 분리

### 2. 고객 문의 대응 현황
- 반복 문의 유형을 모아 FAQ 보강 (진행 중)

→ **향후 방향:** FAQ 보강 완료 후 문의량 변화 확인

## 결정
- 출시 전 발견한 예외 케이스는 별도 항목으로 분리

## Action Items
- [ ] 예외 케이스 처리 항목 생성 @SPEAKER_01 (기한: 6/9)
```
