# 1on1 작성 가이드

> 공통 규칙은 `.claude/skills/meeting/notes-rules.md`를 먼저 적용하세요. 이 파일은 1on1 형식의 본문 골격만 정의합니다.

---

## 무엇을 담나

1on1 회의록은 *다음 1on1에서 이어서 볼 두 당사자*를 위한 기록입니다 — **무엇을 이야기했나(논의 주제) → 서로 무엇을 주고받았나(피드백·요청) → 무엇을 하기로 했나(합의·후속).** 제3자 공유를 전제하지 않는 문서이므로 요점만 담백하게 남깁니다. `## 논의 주제`와 `## 합의·후속`을 제외한 섹션은 **있을 때만** 작성합니다 (`notes-rules.md`의 "빈 섹션 처리" 참고).

---

## 작성 형식

회의록 본문 파일은 `$SESSION_DIR/meeting-notes.md`. 다음 골격을 따르세요:

```markdown
- 날짜: {날짜}
- 참석자: {이름1}, {이름2}    ← meeting.json attendees 기준 평문 (notes-rules.md "참석자 섹션 표기")

## 지난 후속 확인    ← 지난 1on1의 후속을 확인하는 대화가 있었을 때만
- {항목} — {진행/완료/보류}

## 논의 주제

### {주제 1}
- {대화 요지 1~2줄 통합}
- **SPEAKER_XX**: {입장이 갈리거나 요청인 발언}
  - **→ SPEAKER_XX**: {응답}

### {주제 2}
...

## 피드백·요청      ← 주고받은 피드백·요청이 있을 때만
- {피드백 요지} ({SPEAKER_XX})

## 합의·후속
- {합의 사항 또는 다음까지 하기로 한 것}

## Action Items     ← 구체 task가 있을 때만
- [ ] {task} @{담당자}    ← 1인칭 자기지정 @SPEAKER_XX (notes-rules.md 분기)
```

---

## 작성 원칙

- **대화 재현 금지**: 1:1은 발화 왕복이 잦아 발언 단위 기록이 특히 장황해집니다. **주제 단위로 요지만 통합**하고, 발언 불릿(`**SPEAKER_XX**:`)은 입장이 갈리거나 요청-응답이 오간 대목에만.
- **사담은 기록하지 않음**: 안부·잡담·업무와 무관한 한담은 생략. 업무·협업·성장과 닿는 내용만 남깁니다.
- **민감 주제 배려**: 개인사·건강·인간관계 고민 등은 구체 내용 대신 "개인 사정 관련 논의" 수준으로만 남기고, 필요하면 후속 합의만 기록합니다. 회의록 파일은 복사·공유될 수 있음을 전제하세요.
- **합의·후속이 핵심**: 다음 1on1에서 다시 꺼낼 것은 `## 합의·후속`에 반드시 모읍니다. "다음에 다시 얘기하자"고 미룬 주제도 "다음 1on1에서 재논의" 같은 평문으로 여기에 남깁니다 (요약 성격의 섹션이라 결론 태그는 붙이지 않음 — `notes-rules.md` "결론 태그"의 적용 위치 참고).
- **발언 헤더는 SPEAKER 라벨**: 화자 2명이라 매핑이 대개 확실하지만, 발언 헤더는 매핑 확정 여부와 무관하게 `**SPEAKER_XX**:`를 유지합니다 (앱이 표시 시점에 이름 치환).

## 예시 회의록

> 이 유형으로 만들어지는 회의록 샘플입니다 (형식 참고용 — 실제 작성 시 회의 내용으로 채웁니다. SPEAKER_XX 라벨은 앱이 화자 매핑에 따라 실제 이름으로 표시).

```markdown
- 날짜: 2026-06-05
- 참석자: Bobs, Charlie

## 지난 후속 확인
- 코드 리뷰 부담 분산 — 리뷰어 로테이션 도입 후 개선됨 (완료)

## 논의 주제

### 신규 프로젝트 온보딩
- 초반 컨텍스트 파악에 시간이 걸리는 중, 문서보다 페어 세션이 효과적이었음
- **SPEAKER_01**: 다음 주까지는 온전히 적응 기간으로 쓰고 싶다는 요청
  - **→ SPEAKER_02**: 스프린트 목표에서 제외해두겠다고 답변

### 하반기 성장 방향
- 프론트엔드 성능 쪽 전문성을 키우고 싶다는 의사 (SPEAKER_01)

## 피드백·요청
- 회의에서 의견을 더 적극적으로 내줬으면 좋겠다 (SPEAKER_02)

## 합의·후속
- 다음 주 스프린트 목표에서 온보딩 기간 제외
- 성능 스터디 참여 여부는 프로젝트 일정을 본 뒤 다음 1on1에서 재논의

## Action Items
- [ ] 성능 개선 백로그 중 입문용 과제 골라 공유 @SPEAKER_02
```
